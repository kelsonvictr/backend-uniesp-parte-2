Matrícula do Aluno

A entidade que liga aluno e disciplina — com notas e status!

🎯
O que é uma Matrícula?
A tabela que conecta aluno e disciplina
📝 MatriculaAluno — a tabela que conecta tudo!

A matrícula é uma entidade que liga um Aluno a uma Disciplina, com notas e um status. É como o contrato que diz que o aluno Maria está cursando a disciplina de Backend.

Analogia: Matrícula no cursinho

Pensem num cursinho. O aluno existe (cadastro dele), o curso existe (ex: inglês). Mas é a matrícula que liga os dois! A matrícula é um papel que diz "Maria está matriculada no curso de inglês, tirou 8 na primeira prova e 7 na segunda, então foi aprovada". Esse papel é a nossa tabela matricula_aluno.

🗄️ Como vai ficar no banco de dados

🐘 Tabela: matricula_aluno

21 (Maria)2 (Frontend)5.04.0REPROVADO
32 (João)1 (Backend)——MATRICULADO
🔗
Vejam as colunas aluno_id e disciplina_id. São duas chaves estrangeiras! A matrícula "aponta" para o aluno E para a disciplina. É igual ao que vocês já viram com Disciplina → Professor, só que agora temos DOIS @ManyToOne na mesma entidade.
1.1
Enum — MatriculaAlunoStatusEnum
Um "menu fechado" de opções
🏷️ O que é um Enum?

Enum é uma lista fixa de opções. Em vez de usar uma String que pode ser digitada de qualquer jeito (tipo "aprovado", "APROVADO", "Aprovado", ou até "APROVDO" com erro), o Enum garante que só existem as opções que nós definimos.

Analogia: Menu do McDonald's

É como entrar no McDonald's. Você não pode pedir qualquer coisa — só o que tá no cardápio. O Enum é o cardápio fechado do nosso sistema: ou é MATRICULADO, ou é APROVADO, ou é REPROVADO. Não tem outra opção!

package br.com.alunoonline.api.model;

import br.com.alunoonline.api.enums.MatriculaAlunoStatusEnum;
import jakarta.persistence.*;
import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;

@AllArgsConstructor
@NoArgsConstructor
@Data
@Table(name = "matricula_aluno")
@Entity
public class MatriculaAluno {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @ManyToOne
    @JoinColumn(name = "aluno_id")
    private Aluno aluno;

    @ManyToOne
    @JoinColumn(name = "disciplina_id")
    private Disciplina disciplina;

    private Double nota1;

    private Double nota2;

    @Enumerated(EnumType.STRING)
    private MatriculaAlunoStatusEnum status;
}

🏷️ As novidades aqui

@ManyToOne x2
DOIS relacionamentos! Uma matrícula aponta para UM aluno E para UMA disciplina.
@Enumerated(STRING)
Salva o enum como texto legível no banco (ex: "APROVADO"). Sem isso, salvaria como número 0, 1, 2 — confuso!
⚠️
Não esqueçam de adicionar @Enumerated! Se esquecerem, o banco vai salvar 0, 1, 2 em vez de MATRICULADO, APROVADO, REPROVADO. Fica muito difícil de entender depois!

A sacada: o Service define o status inicial!
🎯 Regra de negócio: status inicial

Quando criamos uma matrícula, o status sempre começa como MATRICULADO. O cliente da API não precisa mandar isso — o Service decide. Isso é uma regra de negócio, e é por isso que o Service existe!

package br.com.alunoonline.api.service;

import br.com.alunoonline.api.enums.MatriculaAlunoStatusEnum;
import br.com.alunoonline.api.model.MatriculaAluno;
import br.com.alunoonline.api.repository.MatriculaAlunoRepository;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

@Service
public class MatriculaAlunoService {

    @Autowired
    MatriculaAlunoRepository matriculaAlunoRepository;

    public void criarMatricula(MatriculaAluno matriculaAluno) {
        // O Service define o status inicial automaticamente!
        matriculaAluno.setStatus(MatriculaAlunoStatusEnum.MATRICULADO);
        matriculaAlunoRepository.save(matriculaAluno);
    }
}
💡
Vejam que o cliente só precisa mandar aluno e disciplina no POST. O status é definido pelo Service. Isso é o poder da camada de serviço: regras de negócio ficam escondidas do cliente.

DTO — Data Transfer Object

A peça que transforma nossa API em uma API profissional!

2.1
Primeiro, a dor — depois a solução
Imagine que queremos atualizar APENAS as notas...
🤔 A situação

Uma matrícula já existe. O professor lançou a prova e quer atualizar só as duas notas. Como o cliente (Insomnia) vai mandar esses dados pra nossa API?

Se usarmos o Model MatriculaAluno direto, o cliente teria que mandar ISSO aqui pra atualizar as notas:

😱 Se usássemos MatriculaAluno diretoJSON
{
  "id": 1,
  "aluno": {
    "id": 1,
    "nomeCompleto": "Maria Silva",
    "email": "maria@email.com",
    "cpf": "111.222.333-44"
  },
  "disciplina": {
    "id": 1,
    "nome": "Backend",
    "descricao": "...",
    "cargaHoraria": 80,
    "professor": { ... }
  },
  "nota1": 8.0,      👈 só isso interessa!
  "nota2": 7.5,      👈 e isso!
  "status": "APROVADO"
}
🚨 Os problemas

1. Excesso de dados: o cliente manda um monte de informação que a gente nem vai usar.
2. Feio e confuso: o JSON tá enorme, fica ruim de ler.
3. Risco: e se o cliente mudar o status pra APROVADO sem ter notas? Quem controla?
4. Acoplamento: se mudarmos o Model no futuro, quebramos tudo pro cliente.

2.2
A solução — DTO!
Uma classinha simples pra carregar só o que a gente precisa
✨ Data Transfer Object

Um DTO (Data Transfer Object) é uma classe simples que serve só pra transportar dados entre o cliente e a nossa API. Diferente do Model (que representa a tabela), o DTO representa só o que a gente precisa para uma operação específica.

Analogia: formulário de academia

Imagine que vocês vão na academia e a recepcionista pergunta: "qual sua altura e seu peso?". Vocês não entregam o RG com todos os dados — só falam altura e peso. É isso que um DTO faz: leva só o necessário para a operação.

❌ Sem DTO (Model inteiro)

JSON com 15 campos pra atualizar 2 notas:

{
  "id": 1,
  "aluno": {
    "id": 1,
    "nomeCompleto": "...",
    "email": "...",
    "cpf": "..."
  },
  "disciplina": { ... },
  "nota1": 8.0,
  "nota2": 7.5,
  "status": "..."
}
✅ Com DTO

JSON limpo com só o que importa:

{
  "nota1": 8.0,
  "nota2": 7.5
}
🎯
A regra de ouro: o DTO representa exatamente aquilo que o cliente precisa enviar ou receber, nada a mais, nada a menos. É um "formulário sob medida"!

2.4
O método atualizarNotas — passo a passo
O coração da aula de hoje
🎯 O que esse método precisa fazer?

1️⃣ Buscar a matrícula no banco pelo id
2️⃣ Pegar as notas do DTO e colocar na matrícula
3️⃣ Calcular a média: (nota1 + nota2) / 2
4️⃣ Se média ≥ 7.0 → APROVADO. Senão → REPROVADO
5️⃣ Salvar tudo no banco

public void atualizarNotas(Long id, AtualizarNotasRequestDTO dto) {

    // 1️⃣ Busca a matrícula no banco
    MatriculaAluno matricula = matriculaAlunoRepository.findById(id).get();

    // 2️⃣ Pega as notas do DTO e coloca na matrícula
    matricula.setNota1(dto.getNota1());
    matricula.setNota2(dto.getNota2());

    // 3️⃣ Calcula a média
    Double media = (dto.getNota1() + dto.getNota2()) / 2;

    // 4️⃣ Define o status baseado na média
    if (media >= 7.0) {
        matricula.setStatus(MatriculaAlunoStatusEnum.APROVADO);
    } else {
        matricula.setStatus(MatriculaAlunoStatusEnum.REPROVADO);
    }

    // 5️⃣ Salva tudo no banco
    matriculaAlunoRepository.save(matricula);
}

🔍 Vamos destrinchar cada linha

1️⃣ Buscar a matrícula no banco ▼ 
MatriculaAluno matricula = matriculaAlunoRepository.findById(id).get();

findById(id) vai no banco e busca a matrícula com aquele id. Ele devolve uma caixinha (Optional).

O .get() abre a caixinha e pega o que tá dentro. Pronto, agora temos o objeto matricula pra trabalhar!

2️⃣ Pegar as notas do DTO e colocar na matrícula ▼ 
matricula.setNota1(dto.getNota1());
matricula.setNota2(dto.getNota2());

Aqui acontece a conversão DTO → Model!

Pegamos cada campo do DTO com get e colocamos na matrícula com set. É literalmente passar os dados de um lado pro outro.

Pense: o DTO é o formulário preenchido pelo cliente. A matrícula é o registro oficial no banco. Estamos transcrevendo os dados!

3️⃣ Calcular a média ▼ 
Double media = (dto.getNota1() + dto.getNota2()) / 2;

Matemática básica: soma as duas notas e divide por 2. Se o aluno tirou 8 e 7.5, a média é 7.75.

Guardamos o resultado numa variável media pra usar logo abaixo.

4️⃣ Definir status baseado na média ▼ 
if (media >= 7.0) {
    matricula.setStatus(MatriculaAlunoStatusEnum.APROVADO);
} else {
    matricula.setStatus(MatriculaAlunoStatusEnum.REPROVADO);
}

Regra de negócio! Se média ≥ 7.0, o aluno passa. Se não, reprova.

Vejam que é o Service que decide o status. O cliente nunca manda o status — a API calcula sozinha.

5️⃣ Salvar tudo no banco ▼ 
matriculaAlunoRepository.save(matricula);

save vai no banco e atualiza a linha. Como o objeto matricula já tem id, o Spring é esperto: ele atualiza em vez de criar nova.

🎓
Resumo do que acabou de acontecer: O cliente mandou só 2 notas. A gente buscou a matrícula existente, colocou as notas nela, calculou média, decidiu status, salvou. O cliente não precisou saber de nada disso! Isso é uma API bem feita.

Novidade: PATCH em vez de PUT!
🔧 PUT vs PATCH

PUT = atualiza o recurso inteiro (envia todos os campos).
PATCH = atualiza parcialmente (envia só o que muda).
Para atualizar só as notas, PATCH é o ideal!

package br.com.petshop.api.service;

import br.com.petshop.api.model.Cliente;
import br.com.petshop.api.repository.ClienteRepository;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;

import java.util.List;
import java.util.Optional;

@Service
public class ClienteService {

    @Autowired
    ClienteRepository clienteRepository;

    public void criarCliente(Cliente cliente) {
        clienteRepository.save(cliente);
    }

    public List<Cliente> buscarTodosClientes() {
        return clienteRepository.findAll();
    }

    public Optional<Cliente> buscarClientePorId(Long id) {
        return clienteRepository.findById(id);
    }

    public void deletarClientePorId(Long id) {
        clienteRepository.deleteById(id);
    }

    // ✨ Jeito simples de atualizar!
    public void atualizarClientePorId(Long id, Cliente cliente) {
        cliente.setId(id);             // força o id que veio da URL
        clienteRepository.save(cliente); // save atualiza se id já existe!
    }
}
✨ Por que isso funciona?

O método save do Spring Data é esperto: se o objeto tem um id que já existe no banco, ele atualiza. Se o id não existe ou é nulo, ele cria. Então só precisamos garantir que o id esteja correto — daí o cliente.setId(id) antes de salvar!

💡
Duas linhas pra atualizar. Simples, direto. É assim que muitos projetos de verdade fazem quando o caso é simples. Nada de Optional, nada de isEmpty — só setar o id e salvar.

package br.com.petshop.api.model;

import jakarta.persistence.*;
import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;

@AllArgsConstructor
@NoArgsConstructor
@Data
@Table(name = "pet")
@Entity
public class Pet {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String nome;

    private String raca;

    private Integer idade;

    @ManyToOne
    @JoinColumn(name = "cliente_id")
    private Cliente cliente;
}
